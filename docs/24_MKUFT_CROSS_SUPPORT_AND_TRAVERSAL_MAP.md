Role: turn the public repository from a document collection into a traversable, self-supporting knowledge structure.

---

## 1. Why this document exists

A strong theory can still become difficult to use if its concepts sit beside one another without explicit dependency, routing, and recovery paths.

The public MKUFT repository already contains a canon hierarchy. This document adds the next layer:

> not only what each file is, but how a reader, researcher, or AI system moves between them without losing meaning.

The goal is a self-supporting architecture in which:

- concepts are defined once and reused cleanly;
- every major claim has a layer address;
- formal handles connect to plain-English explanations;
- applications route back to governing invariants;
- falsifiers remain attached to the claims they test;
- a reader can enter from several points and reconstruct the same model.

## 8. Neighbour-link rule

Every major module should identify:

- its parent concept;
- its required mathematical support;
- its nearest application;
- its falsification home;
- the module that limits overreach.

Example:

```text
Ambiguity dynamics
parent: constrained possibility
math support: state-space / transition graph
application: adversarial and capture systems
falsification home: experimental and review discipline
limit: ontology boundary and ordinary explanations
```

This stops modules becoming self-sealed belief systems.

## 9. Duplication rule

Use three levels of repetition:

### Canonical definition

One primary file carries the full definition.

### Compressed reference

Other files carry one or two sentences plus a link.

### Applied translation

Application files restate only the variables and mechanism needed for that domain.

Do not copy large blocks across the repository unless preserving a recovery kernel requires it.

If two files begin carrying different full definitions of the same term, select one canonical home and reduce the other to a routed summary.

## 11. Update protocol

When adding a new concept:

1. Name the problem it solves.
2. Identify whether it is new or an address of an existing invariant.
3. Assign its canonical home.
4. State its S–I–P–O placement.
5. Define variables only where they add discrimination.
6. Link required parent modules.
7. Add predictions and falsifiers.
8. Add the nearest ordinary explanation.
9. Add it to the canon map and README only after the structure is stable.
10. Check that entry from the kernel reconstructs the same meaning.

---

## 12. Coherence audit

A release passes the coherence audit when:

- terms keep the same definition across files;
- equations use compatible variables;
- the same claim is not placed in conflicting layers;
- public and technical explanations point to the same mechanism;
- each application retains an ontology boundary;
- every measurable claim has a falsifier;
- failed submodules can be removed without collapsing unrelated parts.

---

## 13. Hostile-reader audit

Before merge, ask:

- Where does metaphor look like mechanism?
- Where does analogy look like evidence?
- Where is a layer change hidden?
- Where could an observer term become unlimited?
- Where is a mathematical expression only ornamental?
- Which statement lacks a reduction rule?
- Which claim depends on an ontology not established by the data?
- Which simpler explanation has not been tested?

The hostile-reader audit is not surrender to public fashion. It is structural armour.

---

## 14. Compression audit

The repository is well compressed when a reader can recover the same core model from:

- the README;
- the integrated master spine;
- the reconstitution kernel;
- this traversal map;
- a domain-specific application.

The wording may differ. The invariant, layer placement, mechanism, and falsifier should not.
